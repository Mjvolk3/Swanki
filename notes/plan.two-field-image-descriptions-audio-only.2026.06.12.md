## Context

Image flashcards today carry **one** image description per figure — an
interpretive takeaway summary generated by `_generate_image_summary`
(`swanki/processing/image_processor.py:172-283`). That single string is appended
identically to whichever card side the image lands on, and the per-side audio
picker in `swanki/audio/card.py:144-185` speaks it as `Image description: <text>`
on **both** front and back.

The failure surfaced on `alcamoSchaumsOutlineMicrobiology2010_CH03`, card *"Why
is dark-field microscopy useful for observing small or unstained
microorganisms?"*. The figure is on the **front**, so the front audio narrated
the interpretive summary — which states the answer — **before the learner
thinks**. For audio-only learners (AudiobookShelf / BookPlayer), the front audio
is the only way to "see" the figure, so we cannot simply drop the description;
we need a front description that conveys what is *visually present* without
stating or implying the conclusion.

This plan splits the image description into **two fields generated in a single
vision call**:

- **Perceptual** — describes only what is visually in the figure (shapes,
  labels, axes, arrangement). Never states, names, or implies the answer or the
  figure's conclusion. Spoken on the **front**.
- **Interpretive** — the current full takeaway summary. Spoken on the **back**,
  and unchanged everywhere else (reading/lecture landmarks, card-correctness,
  provenance).

Both fields are produced at image-processing time via a Pydantic
structured-output model (the canonical "structure output with models, not
prompts" pattern); placement front/back is decided later in the pipeline, so
both travel to the card and the per-side picker selects which to speak.

**Forward-only.** Existing cards and already-baked audio are untouched — no
migration, no backfill, no re-TTS of old decks. Old serialized records (where
perceptual is absent) deserialize fine and fall back to current behavior.

## Key Design Decisions

1. **Generate both fields in ONE vision call via a new `ImageDescription`
   Pydantic model** (`perceptual: str`, `interpretive: str`). Why: one image,
   one model invocation — half the API cost/latency of two calls, and the model
   sees the figure once and writes both views coherently. This is the canonical
   "structure output with Pydantic models, not by tweaking prompts" case from
   `CLAUDE.md`. The model lives in `swanki/models/document.py` (next to
   `ImageSummary`), imported by `agents.py`.

2. **Register `image_description_agent` in `swanki/llm/agents.py` following the
   existing one-agent-per-output-type pattern**:
   `image_description_agent: Agent[None, ImageDescription] = Agent(output_type=ImageDescription, retries=3)`.
   Call it through the **existing** `with_safety_retry(image_description_agent,
   [prompt, image_content], model=..., model_settings=..., label="image
   description")`. Why: `with_safety_retry` is output-type-agnostic
   (`safety.py:81-95` docstring confirms it works with any Agent regardless of
   output type) and returns the full `RunResult`, so `result.output` is the typed
   `ImageDescription`. `_augment_with_preamble` (`safety.py:43-68`) prepends the
   educational preamble to the first `str` in the list and leaves the image
   untouched — biosec-refusal retry keeps working unchanged.

3. **Keep `CardContent.image_summary` (cards.py L92-94) as the INTERPRETIVE/back
   text, add one new optional `image_summary_perceptual: str | None`.** Why:
   minimal blast radius. Every current consumer (`card_correctness.py`,
   provenance serialization, the existing back-audio path) expects
   `image_summary` to be the interpretive text — renaming it would ripple
   through all of them. The new field is additive and optional.

4. **On `ImageSummary` (document.py L52-57) keep `summary` as interpretive, add
   `perceptual: str | None = Field(None, ...)`.** Why: optional/defaulted so old
   serialized `ImageSummary` records (in `provenance.yaml`, run artifacts) still
   deserialize. `ImageSummary` has `extra="forbid"` (L50) but adding a known
   field is safe — only *unknown* keys are rejected.

5. **Rejected: computed-property aliasing** (e.g. a `@property` mapping
   `interpretive -> summary`). Why: it hides the dual-field shape from
   serialization and from readers, and Pydantic computed fields complicate
   round-tripping. Two plain `Field`s are explicit and serialize cleanly.

6. **Rejected: a second-pass LLM "leakage gate"** that re-checks the perceptual
   text for answer leakage. Why: scope creep and added cost/latency per figure.
   Leakage is controlled at the source — a strong `Field(description=...)` on
   `perceptual` plus an explicit prompt rule ("describe only what is visually
   present; never state, name, or imply the answer or the figure's conclusion")
   — plus a manual eyeball of a few generated cards during verification.

7. **Rejected: wiring the dual-field prompt to the unused
   `default.yaml`/`book_voice.yaml` `image_summary` config keys.** Why: those
   keys are currently dead; activating them widens scope. The dual-field prompt
   is authored directly at the `_generate_image_summary` call site, consistent
   with current behavior. We MAY borrow the perceptual phrasing already in
   `default.yaml` ("describe only what is shown, do not interpret").

8. **The split is CONFINED to card front/back audio.** `_insert_image_summary`
   (image_processor.py L285-299), the reading-landmark path (reading.py
   L218-225), and the lecture path (lecture.py L530-537) all CONTINUE to consume
   the INTERPRETIVE summary. Why: a bare perceptual description ("a bright object
   on a dark background") makes a useless navigational `Figure:` landmark — the
   landmark must carry the takeaway.

9. **Graceful fallback in the picker:** if `image_summary_perceptual` is absent
   (old cards, generation gap), the front falls back to `image_summary`
   (interpretive) — i.e. exactly current behavior. Why: forward-only guarantee —
   nothing regresses for pre-existing cards.

## Approach

Execution order (each step is independently testable):

**1. Model fields.** In `swanki/models/cards.py`, add after `image_summary`
(L92-94):

```python
image_summary_perceptual: str | None = Field(
    None,
    description="Perceptual description: only what is visually present in the "
    "figure; never states or implies the answer. Spoken on the card front.",
)
```

In `swanki/models/document.py`, on `ImageSummary`, add `perceptual: str | None =
Field(None, description="Perceptual (front-audio) description; optional for "
"backward compat")`. `summary` stays the interpretive description.

**2. `ImageDescription` model + agent.** In `swanki/models/document.py` add the
structured-output model (single short definition):

```python
class ImageDescription(BaseModel):
    """Dual image description from one vision call."""
    perceptual: str = Field(description="Only what is visually present in the "
        "figure (shapes, labels, axes, arrangement). NEVER state, name, or "
        "imply the answer or the figure's conclusion.")
    interpretive: str = Field(description="Full takeaway summary: what the "
        "figure shows and means, 2-4 sentences.")
```

In `swanki/llm/agents.py`, import `ImageDescription` and register
`image_description_agent: Agent[None, ImageDescription] = Agent(output_type=ImageDescription, retries=3)`.

**3. Dual-field vision call.** In `image_processor.py::_generate_image_summary`
(L172-283): change the prompt to request both a perceptual and an interpretive
description (carry the existing context + the explicit non-leaking rule), swap
`with_safety_retry(text_agent, ...)` (L274-280) for
`with_safety_retry(image_description_agent, ...)`. `result.output` is now an
`ImageDescription`. Change the return type from `str | None` to
`ImageDescription | None` (or a `(interpretive, perceptual)` tuple). At the call
site (L88-96) store both: `image_info["summary"] = desc.interpretive` and
`image_info["summary_perceptual"] = desc.perceptual`. Keep writing the
interpretive text to the `image_summaries_dir` sidecar (L93-95) so downstream
landmark/telemetry stays interpretive.

**4. Pipeline carries both.** In `pipeline.py::process_images` (L737-789),
construct `ImageSummary(..., summary=info["summary"],
perceptual=info.get("summary_perceptual"))`. In the image-card assignment block
(L1316-1331) capture `image_summary_perceptual_text = img_summary.perceptual`
alongside `image_summary_text = img_summary.summary`. In the placement branches
(L1437-1509), wherever `card.front.image_summary = image_summary_text` /
`card.back.image_summary = ...` is set, also set
`card.front.image_summary_perceptual = image_summary_perceptual_text` (front
branches) and `card.back.image_summary_perceptual = ...` (back branches). The
existing "skip card if no interpretive summary" guard is unchanged — perceptual
is best-effort.

**5. Per-side audio picker + fallback.** In `swanki/audio/card.py` (L154-169):
- Front branch (`is_front`, has `front.image_path`): use
  `card.front.image_summary_perceptual` **if present**, else fall back to
  `card.front.image_summary` (current behavior).
- Back branch: use `card.back.image_summary` (interpretive) — unchanged.

The `Image description: <text>` append (L181-182) stays as-is; only the *source*
string differs by side.

**6. Front system-prompt wording.** In
`card.py::_build_transcript_system_prompt` (L807-885), the front-branch image
instructions (L816-824 cloze, L843-851 non-cloze) currently imply the
description is answer-bearing ("crucial for audio-only learners"). Soften to:
the front image description describes only what is visible and does NOT answer
the question — keep reading it verbatim. Minor wording change only; the back
branch (L910-921) is untouched.

**7. Reading/lecture stay interpretive.** Verify (no code change) that
`reading.py` (L218-225) and `lecture.py` (L530-537) consume the `Figure:`
landmark built from interpretive `summary`/`first_sentence(summary)`
(`process_tables` at pipeline.py L812 uses `img.summary`), and that
`_insert_image_summary` (L285-299) uses `image_info['summary']`. None reference
`perceptual`.

## Gotchas

1. **Don't `.strip()` the `RunResult`.** `result.output` is now a typed
   `ImageDescription` (see Decision #2); `.strip()` applies to
   `desc.perceptual` / `desc.interpretive` individually.

2. **Reading/lecture landmarks must NOT switch to perceptual** (Decision #8).
   The easiest accidental regression — double-check no stray edit points
   `process_tables`/`reading`/`lecture`/`_insert_image_summary` at `perceptual`.

3. **Leakage: if a generated perceptual still leaks, tighten the prompt
   wording — do not add a runtime gate** (Decision #6).

4. **`ImageSummary` has `extra="forbid"` (document.py:50).** Adding the known
   field `perceptual` is fine; this guard only rejects *unknown* keys, so old
   records without `perceptual` still parse (the field defaults to `None`).
